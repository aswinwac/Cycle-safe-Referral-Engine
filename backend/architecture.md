# Architectural Note

## Overview
The Cycle-Safe Referral Engine uses a distributed system architecture to efficiently validate graphs (DAGs), store relational data, and orchestrate asynchronous workflows while remaining highly available and observable.

## Key Components

1. **FastAPI (API Gateway / Presentation Layer)**
   - Exposes RESTful endpoints for the dashboard, fraud analytics, rewards generation, and core user actions.
   - Pydantic ensures strong type validation and immediately generates OpenAPI (Swagger) Documentation available at `/api/v1/docs`.

2. **PostgreSQL (System of Record)**
   - Stores authoritative states for Users, Referral events, Fraud Signals, and general Activity.
   - Designed around append-only and idempotent structures (e.g. `RewardRecords` matching `referral_id + level`).
   - Powers the core Dashboard API through fast aggregated queries (`SELECT count(*) FILTER...`).

3. **Neo4j (Graph Processing)**
   - Maintains a secondary structural truth of the referral DAG.
   - Directly utilized by the Python `ReferralService` utilizing Cypher queries to recursively walk ancestral lines and actively block cycles pre-insertion in **< 20ms**, ensuring cycles never enter PostgreSQL.
   - Connects directly to the frontend's visual network topology explorer.

4. **Redis (Caching & Rate Limiting)**
   - Sliding window algorithm tracks attempt frequency to catch automated abuse.
   - Blocks users attempting to exceed their allocation or brute-force validations.
   - Caches computationally heavy items like Neo4j descendant sub-trees.

5. **React.js Dashboard (Client)**
   - Consumes live endpoints.
   - Renders Recharts for metrics velocity and `react-force-graph-2d` for real-time visualization of the network topology generated by Neo4j graph data.

## Deployment
All services are containerized via Docker and orchestrated through `docker-compose`, making configuration reproducible, segregated, and scalable seamlessly into clouds like AWS / GCP using tools like Kubernetes or native clusters.
